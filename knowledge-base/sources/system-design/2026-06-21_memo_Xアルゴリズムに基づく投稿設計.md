# Xアルゴリズムに基づく投稿設計

## 要約
X（旧Twitter）が公開している For You 表示アルゴリズムのリポジトリ構造をもとに、「アルゴリズムに好かれる投稿」ではなく **「特定ユーザーの過去行動と近く、ポジティブ行動を起こしやすく、ネガティブ行動を起こしにくい投稿」** を作る、という投稿設計の指針をまとめたもの。For You feed は In-Network（フォロー中）と Out-of-Network（フォロー外）の投稿を統合し、Phoenix でランキングする。投稿時に気にすべき点は、リポジトリ上の主に5つの仕組み —— ① engagement history、② candidate hydration、③ embeddings / similarity search、④ multi-action prediction、⑤ filters / negative actions —— に紐づく。

## 学び・発見
- **Xは「単一の良し悪しスコア」では評価しない。** Phoenix は favorite / reply / repost / quote / click / dwell / follow_author などポジティブ行動の確率と、not_interested / mute / block / report などネガティブ行動の確率を**個別に予測**し、重み付き合計で最終スコアを出す。だから「何を狙う投稿か」を事前に決めることが効く。
- **フォロー外配信が前提なので、投稿は「誰の過去行動に近いか」と「初見で意味が通るか」の両方が要る。** Out-of-Network 候補は Phoenix Retrieval がグローバルコーパスから埋め込み類似度で拾うため、文脈の薄い告知（例：「新刊発売！」）は弱い。
- **ネガティブ行動もランキング対象。** 過度な煽り・釣り・文脈なしの連投は、クリックは取れてもミュート/ブロック/通報を誘発し、負の重みで長期的にスコアを下げる。これは精神論ではなくリポジトリ上の構造。

## 投稿チェックリスト（Xリポジトリ上の根拠つき）

| 投稿時に気にすること | なぜ重要か | リポジトリ内で関係する情報 |
|---|---|---|
| 1. 誰向けの投稿か明確か | フォロー外投稿はユーザー履歴と投稿埋め込みの近さで候補化される | Phoenix Retrieval の Two-Tower Model。User Tower がユーザー特徴・engagement history を埋め込み化、Candidate Tower が投稿を埋め込み化し、dot product similarity で候補取得 |
| 2. 初見でも意味が分かるか | フォロー外前提なので、投稿単体で文脈が伝わらないと click・dwell・like 等の予測が弱くなる | Phoenix は P(favorite), P(reply), P(repost), P(click), P(dwell) など複数行動を予測。最終スコアは各確率の重み付き合計 |
| 3. 何が面白い/価値あるか伝わるか | 内容が薄いとユーザー履歴との関連性をモデルが拾いにくい | 候補投稿は Hydration で text, media, author information, video duration などを補完。本文・メディア・投稿者情報が評価材料 |
| 4. 過去に反応していそうな文脈が入っているか | Xは過去反応を見て「この人に関連する投稿か」を判断する | Grok-based transformer が liked, replied, shared などの engagement history を理解して関連コンテンツを判断（README） |
| 5. 1文目で止まる理由があるか | dwell が予測対象なので、止まって読まれること自体が評価材料 | Phoenix の予測対象に P(dwell)。example user history にも dwell が action として定義 |
| 6. いいね/RT/クリック/滞在のどれを狙うか決まっているか | 単一スコアでなく複数行動ごとに確率予測しているため | Phoenix は favorite, reply, repost, quote, click, profile_click, video_view, photo_expand, share, dwell, follow_author などを予測 |
| 7. 釣りすぎてミュート・興味なしを生まないか | ネガティブ行動も予測され最終スコアを下げる | 予測対象に P(not_interested), P(block_author), P(mute_author), P(report)。ネガティブ行動は負の重みでコンテンツを押し下げる（README） |
| 8. 画像・動画・引用・スレッド化した方が伝わらないか | メディア有無・動画視聴・画像拡大も評価対象 | Candidate Hydration で media entities / media detection / video duration を補完。予測対象に P(video_view), P(photo_expand) |
| 9. 告知だけでなく感情・発見・比較・読者像が入っているか | 手作業特徴量でなくユーザー履歴と投稿内容の関連性を学習するため、文脈が濃い方が「誰に近いか」が出やすい | hand-engineered feature を排除し、Grok-based transformer が engagement sequence から関連性を学習（README） |
| 10. フォロー外の人にも伝わるか | Out-of-Network 候補としてグローバルコーパスから拾われる | For You feed は In-Network と Out-of-Network を統合し、Out-of-Network は Phoenix Retrieval が global corpus から発見 |

## 実務寄りの要点（仕組み別）

1. **誰向けか（Retrieval）** — フォロー外投稿は Phoenix Retrieval が探す。User Tower（ユーザー特徴・過去反応を埋め込み化）と Candidate Tower（投稿を埋め込み化）の dot product similarity で近い投稿を取る構造。投稿側で大事なのは「誰の過去行動と近い投稿なのか」がはっきりしていること。告知ワード単体より、対象読者の文脈語（テーマ・主人公像・ジャンル）が入っている方が特徴が立つ。

2. **初見で意味が分かるか（Hydration/Scoring）** — For You は In-Network と Out-of-Network を混ぜるため、見る人は作者を知らない前提。初見で意味が分からない投稿は click・dwell・favorite・reply に繋がらず、Phoenix の各行動予測が弱くなる。投稿単体で「何の話か・誰向けか」が伝わるよう書く。

3. **1文目で止まる理由（dwell）** — Phoenix の予測対象に P(dwell) があり、ユーザー履歴の action にも favorite / reply / quote / repost / dwell / video quality view が定義されている。つまり「滞在」自体が評価材料。一瞬で流されず、止まって読まれる導入が要る。（ただし dwell が秒数や表示%でどう定義されるかは README からは断定できない。）

4. **どの反応を狙うか（multi-action prediction）** — X は「いいね数」だけを見ない。Phoenix は P(favorite), P(reply), P(repost), P(quote), P(click), P(profile_click), P(video_view), P(photo_expand), P(share), P(dwell), P(follow_author) などを個別予測する。投稿前に狙いを決めると設計しやすい：認知拡大→repost・quote・share／作品理解→dwell・photo_expand・click／購入誘導→click・profile_click／ファン化→follow_author・reply。

5. **釣りすぎない（negative actions）** — ポジティブ行動だけでなく P(not_interested), P(block_author), P(mute_author), P(report) も明示され、負の重みでコンテンツを押し下げる。クリックは取れても興味なし/ミュート/ブロック/通報を生む投稿は長期的に危険。過度な煽り・誤解を誘う釣り・対象外への広げすぎ・文脈なし連投は避ける。

## 最終的な判断軸（投稿前チェックの順序）
1. **Retrieval に乗るか** — 誰の興味ベクトルと近い投稿か
2. **Hydration で特徴が立つか** — 本文・メディア・投稿者・言語・引用・動画が分かりやすいか
3. **Scoring で良い予測が出そうか** — like / repost / reply / click / dwell / follow につながるか
4. **Negative prediction を避けられるか** — not_interested / mute / block / report を誘発しないか
5. **Filtering で落ちないか** — 重複・古すぎ・ミュートワード・スパム・暴力グロ等に該当しないか

> 結論：「アルゴリズムに好かれる投稿」を作るのではなく、「特定ユーザーの過去行動と近く、ポジティブ行動を起こしやすく、ネガティブ行動を起こしにくい投稿」を作る、という理解が最も近い。
