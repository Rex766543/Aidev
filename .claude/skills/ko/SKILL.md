# ナレッジ整理スキル (ko)

## 概要
X投稿・Web記事・PDF・スライド・音声入力・手元メモなどのソースを受け取り、
RAG / AI が後から参照しやすい構造化 Markdown に整理して `knowledge-base/sources/_inbox/` に出力する。
複数ソースをまとめて1ファイルに整理する場合もある。
このスキルは**整理のみ**を行い、カテゴリ振り分けは行わない（それは `/cat` の役割）。

## トリガー
- `/ko` コマンドから呼び出される
- 直接スキルとしても利用可能

## 入力
- ソース本体（貼り付けテキスト / URL / ファイルパス / PDF / 音声ファイル / 音声書き起こし / メモ）。複数同時もあり得る。
- 任意の「まとめ観点」指定（例: 「コスト面だけ」「反論材料として」）。指定があれば必ずそれに従う。

## URL入力の取得（前処理）

入力にURLが含まれる場合、整理に入る前に本文テキストを確定させる。
**取得手段の選択ルールと Jina の能力・制約は `.claude/skills/_shared/jina-reader.md` を参照する**

要点だけ:
- **X/Twitter URL** → Jina (`jina_read_url`)。失敗時はユーザーに報告し、テキスト貼り付けを促す（無理に回避しない）。`source_type: "x"`
- **通常URL** → WebFetch（標準）。失敗時のみ Jina にフォールバック。`source_type: "article"`
- 取得した本文を以降の整理フローの入力とする。

## 音声ファイルの前処理（Whisper 文字起こし）

入力が音声ファイル（拡張子 `.mp3` / `.wav` / `.m4a` / `.aac` / `.flac` / `.mp4` / `.ogg` など）の場合、
まず Whisper でテキスト化してから通常の整理フローに進む。

### 手順

```bash
# 1. 一時出力先を決める
OUT_JSON="/tmp/ko_whisper_$(date +%s).json"

# 2. Whisper で文字起こし（medium モデルを使用）
/Users/ishikawaryuusuke/Desktop/aidev/TravelCreator/subtitle-pipeline/.venv-whisper/bin/python \
  /Users/ishikawaryuusuke/Desktop/aidev/tools/whisper/transcribe_local.py \
  "<音声ファイルの絶対パス>" \
  "$OUT_JSON" \
  medium
```

- モデルはデフォルト `medium`。ユーザーが `large` や `large-v3` を指定した場合はそちらを使う。
- 完了後、Read ツールで `$OUT_JSON` を読み込み、`.text` フィールドを本文テキストとして扱う。
- `source_type` は `"audio"` にする。
- 文字起こし原文はナレッジ本文末尾に `## 文字起こし原文` セクションとして付記する（後で検索可能にするため）。

## PPTXファイルの前処理（markitdown テキスト化）

入力がスライドファイル（拡張子 `.pptx` / `.ppt`）の場合、
まず markitdown でテキスト化してから通常の整理フローに進む。

```bash
python -m markitdown "<ファイルの絶対パス>"
```

- `source_type` は `"slide"` にする。
- markitdown が失敗した場合はユーザーに報告し、テキスト貼り付けを促す。

## 大原則（トークン効率と品質）
このナレッジは将来 AI / RAG が検索して参照する。AI が**必要なファイルだけ**を開き、
**不要なコンテキストを取り込まず**高品質な出力ができることが最優先。そのため:

1. **まずソースの本論（主張・核心）を分析する。** 本論に直接関わらない枝葉・装飾・余談は整理から省く。
   - 記載すべきは「本論を理解・活用するのに必要な情報」だけ。
2. **出力は最低限の分量に絞る。** 冗長な言い換え・原文の逐語コピー・自明な説明はしない。
   要点を圧縮して書く。長さではなく密度で勝負する。
3. **専門知識の補完。** ソース内の情報だけでは専門家以外が（または将来の AI が文脈なしに）理解できない
   論理・前提に依存している場合、ソースに無い背景知識を簡潔に補う。補った箇所は事実と区別できるよう
   `（補足）` を付す。
4. **観点指定があるときは、その観点に資する情報だけを抽出・整理する。** 膨大なソースを全網羅しない。

## メタデータ規約（最小構成）
冒頭に YAML frontmatter を置く。AI が relevance 判定・引用に使う項目のみ。余計な項目は足さない。

```yaml
---
title: "<ファイル名と一致する簡易タイトル>"
source_type: "x" | "article" | "pdf" | "slide" | "audio" | "memo"
source_url: "<URL。無ければこの行を省略>"
captured_at: "YYYY-MM-DD"   # 本日の日付
topic:
  - <推定カテゴリ。確信が持てなければ仮置きでよい。/cat が最終決定する>
tags:
  - <検索軸となる固有名詞・概念を複数。具体的に>
---
```

- そのソース固有の理由で項目が要るときだけ、判断して追加する。

## ファイル命名
`YYYY-MM-DD_{source_type}_{短いタイトル}.md`
（例: `2026-06-13_x_委託制度と過剰流通.md`）
- 日付は本日。
- frontmatter の `title`・本文 H1・ファイル名（拡張子除く）を一致させる。

## 本文フォーマット

### 必須セクション（順序固定）
1. `# <タイトル>` — frontmatter の title と一致。
2. `## 要約` — ソース全体の主要ポイントを網羅した要約。複数文。これを読めば本論が掴める粒度。
3. `## 学び・発見` — このソースから得た本質的な学び。**数を絞る**（ソース全体を俯瞰して「特に言いたいこと」だけ）。

### 必須セクションより後（可変）
セクション項目は固定しない。**毎回、そのソース内容に応じて最も分かりやすい構造を都度考えて整理する。**
これを可変にしている目的は、固定の項目に機械的に当てはめることではなく、内容に応じて分かりやすく
整理するため。出力を最低限の分量に絞る方針はここでも同じ。

## 出力先と完了後の動作
1. 整理結果を `knowledge-base/sources/_inbox/{命名規則のファイル名}.md` に書き出す。
2. 何を作ったか（ファイル名・要約・付与タグ）を簡潔に報告する。
3. **AskUserQuestion で「続けてカテゴリ分けの `/cat` を実行しますか？」を確認する。**
   - 「はい」→ `/cat` スキル（`.claude/skills/cat/SKILL.md`）を、今作成したファイルを対象に実行。
   - 「いいえ」→ ファイルは `_inbox/` に残す旨を伝えて終了。
   - `/ko` と `/cat` を分けているのは、`/cat` を単体でも使えるようにするため。
