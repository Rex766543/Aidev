# アドホック課題資料インプット

## タイトル
負荷テスト検知エラーを受けたアーキテクチャ改善提案

## クライアント/サービス名
platform-team

## 説明対象・ターゲット
エンジニアリングマネージャー・CTOクラス（技術背景あり）

## 課題の概要・背景
直近の負荷テスト（2026年2月実施）において、同時接続数500超でAPIレスポンスタイムのP99が5秒を超過するエラーを検知。
現行のモノリシック構成でDBへの負荷が集中し、スケーラビリティの限界が露呈した。
本番環境での再現リスクが高く、早急なアーキテクチャ対応が必要。

## 分析・調査結果
- エラー発生箇所: 検索API・ランキング集計API の2エンドポイントに集中
- ボトルネック: 複数のAPIが同一DBに同期クエリを発行 → コネクションプールが枯渇
- P99レイテンシ推移: 通常時 120ms → 500同時接続時 5,400ms（約45倍）
- DBのCPU使用率: ピーク時 94%（閾値80%超で警告）
- 過去の検討（2025年10月）でもキャッシュ導入案が挙がっていたが未着手
- 参考: `platform-team/references/architecture-review-2025Q3.md` に前回検討の詳細あり

## 対応案

案A（推奨）: キャッシュ層（Redis）導入 + 非同期キュー化
- 読み取り系APIはRedisキャッシュ経由に切り替え
- 集計処理はMessageQueue（SQS等）で非同期化
- DBへの直接クエリを最小化

案B: 読み書きDBの分離（レプリカ構成）
- Read Replicaを追加し、参照クエリをオフロード
- 構成変更のみで済むが、根本的なスケーラビリティ改善には限界あり

案C: マイクロサービス分割
- 検索・ランキングを独立サービスとして切り出し
- 長期的には最善だが、移行コスト・工数が大きく短期対応には不適

## 推奨案と理由
案Aを推奨。
- 改修範囲が限定的で本番影響を抑えつつ即効性が高い
- Redisは社内実績あり（認証基盤で使用中）
- 非同期化により将来的なスケールアップにも対応可能
- 案Bとの組み合わせも可能（段階的移行）

## アクションプラン
- Week1-2: Redis導入・設計レビュー（インフラ・バックエンドチーム）
- Week3-4: キャッシュ層実装・単体テスト
- Week5: ステージング環境での負荷テスト再実施（同条件）
- Week6: 本番デプロイ・モニタリング強化
- 目標: P99レイテンシ 500ms以内（現行比90%改善）
